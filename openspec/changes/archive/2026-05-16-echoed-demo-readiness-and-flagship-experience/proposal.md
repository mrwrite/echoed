# EchoEd Demo Readiness and Flagship Experience

## Why

EchoEd already demonstrates strong architectural direction across governed delivery, curriculum quality, assessment evidence, mastery, runtime guidance, staff governance visibility, organizations, and role-aware dashboards. What it does not yet guarantee is a stable, polished, repeatable demo environment that can be reset and presented confidently without manual preparation.

The current platform is advanced enough to impress in isolated flows, but live demos still depend too much on seed assumptions, incidental data state, and operator memory. That creates avoidable presentation risk. A strong product demo needs deterministic orgs, users, course content, progression states, mastery states, governance examples, runtime recommendation examples, and a safe reset path.

This change exists to make EchoEd reliably demoable around the flagship “Introduction to Africa” experience. It does not expand the platform’s educational engines. It makes the existing platform reproducible, presentation-ready, and operationally dependable for live product walkthroughs.

## What Changes

- Define a deterministic demo environment with stable seeded organization, admin, teacher, student, section, enrollment, course, unit, lesson, assessment, mastery, and intervention data
- Stabilize the flagship “Introduction to Africa” pathway so student, teacher, and admin flows always show predictable states
- Seed bounded learner archetypes that reliably demonstrate mastered, struggling, ambiguous, and normal progression behavior
- Ensure seeded evidence and governance conditions also produce predictable publish-readiness, safe-publish, lineage-safety, and competency-integrity examples
- Add a documented, repeatable demo reset and setup path that is safe to rerun
- Define a polish plan for the flagship presentation experience without redesigning the UI in this change
- Improve demo operational readiness so live walkthroughs do not require manual data repair

## Explicit Non-Goals

- No new governance engines
- No new versioning systems
- No CMS rewrite
- No AI workflow orchestration
- No live classroom synchronization
- No complex analytics platform
- No automated interventions
- No bypassed auth or demo-only permission shortcuts

## Impact

- Backend and Demo Data:
  - Strengthen seeded org, user, course, progression, mastery, intervention, and governance states
  - Reuse existing delivery, scoring, mastery, runtime, governance, certification, and reporting systems rather than creating demo-only logic
  - Add deterministic reset and setup expectations for repeatable demos

- Frontend and UX Planning:
  - Keep current UI behavior intact in this change
  - Define where flagship polish work should land next without redesigning current surfaces yet

- Operations:
  - Reduce live-demo fragility
  - Make demo startup and reseed behavior safer and more predictable
  - Improve confidence in repeated stakeholder walkthroughs

- Strategic Alignment:
  - Supports the flagship “Introduction to Africa” proving-ground pathway
  - Reinforces the platform’s governed learning, staff visibility, and runtime guidance story through stable seeded states
